And why is *that* important? Well, look again at whatâ€™s on that element. Weâ€™ve also got `@input="updateValue"`. That means we have two `input` event listeners. Okay so we know what caused this funky behavior. But how do we fix it? Take a few deep breaths, because the solution to this problem is as convoluted as the issue that caused it. ğŸ§˜â€â™€ï¸ğŸ§˜â€â™‚ï¸

## Resolving Our Conflict

If two people are in conflict about something, whatâ€™s a good way to resolve that? How about they shake hands and compromise? What does this have to do with our code? Well, thatâ€™s basically what weâ€™re about to implement: a compromise. We can encapsulate that compromise within a computed property, which weâ€™ll call `listeners`.

ğŸ“ƒ **src/components/BaseInput.vue**
```javascript

      listeners() {
        return {
          ...this.$listeners,
          input: this.updateValue
        }
      }
    }
```
Since both of our `input` events want to exist but theyâ€™re currently messing with each other, the compromise is: they can both exist alongside each other within a `listeners` object, which our computed property returns. Notice how weâ€™re using the object spread operator to merge the inherited `$listeners` into this object (i.e. the `input` event listener hidden in `v-model`), and below that is a property for our `input` event that triggers the `updateValue` method.

Since there are now two `input` definitions on this object. Because of the way JavaScript objects work, the one that is defined last will take precedence. In this case, the `input` event that triggers our `updateValue` method takes precedence, which solves our issue!

Now, in the template, we can simply add our computed `listeners` to the `input` element.

ğŸ“ƒ **src/components/BaseInput.vue**
```html

```
